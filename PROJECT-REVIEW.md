# Coach Titan MVP 项目复盘（面试向）

## 1. 项目定位
- 目标用户：健身爱好者（增肌/减脂/维持），不想精确计算但希望有可用指引的人群
- 核心痛点：记录复杂、心理负担重、难以坚持
- 价值主张：用“拳掌法 + 拍照估算”降低记录成本、保持可执行性

## 2. 关键假设（待验证）
- 用户愿意用拍照替代精确称重记录
- 双角度 + 参照物能提高可信度而不会显著降低完成率
- 简短可执行的反馈优于精确数值

## 3. 关键迭代与思路

- 验证：拍照完成率、识图满意度（1-5 分）
- 影响：待验证

## 4. 当前结论（阶段性）
- 拍照路径可行，但需要更清晰的拍摄指引与反馈提示
- 用户对“可信度”非常敏感，文案与解释会显著影响接受度
- 目前尚缺少用户数据，验证体系需要尽快补齐

## 5. 下一步计划（1-2 周）
- 设计单角度 vs 双角度的完成率对比测试
- 增加拍照后“可信度评分 + 反馈按钮”
- 收集 5-10 位用户的首次体验反馈

## 6. 面试说明话术（可直接使用）
- 我把 MVP 当作验证实验来做：每次更新都会明确目标、动作、验证方法与预期影响。
- 目前聚焦在“降低记录成本”的核心价值，通过拍照估算替代精确计算。
- 下一步重点是完成率与可信度验证，用数据决定是否保留双角度流程。

## 7. MVP Demo Script（1 分钟）
- 路径：Home → 饮食输入 → 生成策略 → 日志复盘 → 备份导出
- 卖点 1：记录成本低（拍照/手动输入 + 拳掌法）
- 卖点 2：可解释复盘（结构化建议 + 最近记录回看）
- 卖点 3：数据可控（本地存储 + 导出备份）
- 风险提示 1：AI 识别存在误差，结果仅供参考
- 风险提示 2：非医疗/营养处方，需结合专业建议

## 8. 演示前自检清单
- 运行 `npm run build`，确保可正常打包
- 运行 `npm run dev`，首页正常展示（无空白/报错）
- 完整走一遍主流程：记录 → 建议 → 日志 → 备份导出
- 备份导入演示：导出后导入（覆盖提示正常、导入成功提示正常）
- 空状态检查：无历史记录时提示文案清晰
- AI 失败降级：断网或缺 `ARK_API_KEY` 时仍可手动保存与回看

## 10. 营养估算依据与边界
- 依据：拳掌法为营养教练常用的快速估算方法；基础代谢采用常见 Mifflin-St Jeor 公式 + 性别系数 + 训练频率活动系数
- 适用范围：面向一般健康人群的粗略估算，用于建立结构化饮食习惯
- 限制：特殊生理状态未单独建模；性别/身高/体重/年龄缺失时仅使用拳掌法估算
- 不适用人群：孕期、进食障碍、临床疾病或受伤恢复期人群
- 结论：仅供参考，不构成医疗/营养处方
